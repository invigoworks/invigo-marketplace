# 연관 문서 교차 참조 규칙

> **목적**: 기획서 작성/수정 시 연관된 기획 문서를 체계적으로 식별하고, 문서 간 충돌과 누락을 사전 방지한다.

---

## 1. 연관 문서 식별 기준 (7가지)

| # | 식별 기준 | 설명 | 예시 |
|---|----------|------|------|
| 1 | **공유 엔티티** | 동일한 데이터 엔티티(테이블/타입)를 참조하는 문서 | 재고 조정, 재고 이동이 모두 `Inventory` 엔티티 사용 |
| 2 | **화면 네비게이션** | 기획서의 "연결 화면"에 명시된 FROM/TO/RELATED 관계 | 판매주문서 → 출고 페이지 |
| 3 | **상태 전이 체인** | 한 문서의 출력 상태가 다른 문서의 입력 조건이 되는 관계 | 발주(CONFIRMED) → 입고(PENDING) |
| 4 | **권한 매트릭스** | 동일한 권한 코드를 사용하거나 동일 역할의 권한 범위가 겹치는 문서 | 재고 조정과 재고 이동이 동일 `inventory:*` 권한 사용 |
| 5 | **API 공유** | 동일 API 엔드포인트를 호출하는 문서들 | 재고 현황과 재고 조정이 동일 `GET /inventory` API 사용 |
| 6 | **기능 게이트** | 한 기능이 다른 기능의 선행 조건인 관계 | 거래처 등록 → 판매주문서 작성 시 거래처 선택 |
| 7 | **데이터 파생** | 한 문서의 데이터가 다른 문서에서 계산/집계되어 표시되는 관계 | 개별 입출고 → 재고 현황 집계 |

## 2. 연관 문서 검색 방법

### 2.1 검색 우선순위

```
1단계: 매니페스트 검색 (0 토큰, 즉시)
   └─ .claude/shared-references/notion-manifest.md에서 키워드로 관련 문서 식별

2단계: plan-chatbot RAG 검색 (저비용)
   └─ ChromaDB 벡터 검색으로 의미 기반 연관 문서 발견

3단계: Notion MCP 조회 (고비용, 필요시만)
   └─ 1-2단계로 식별된 문서의 상세 내용 확인
   └─ ⚠️ 3건 이상 fetch 시 반드시 subagent 격리
```

### 2.2 매니페스트 기반 검색

```bash
# 매니페스트에서 키워드로 연관 문서 식별
# 예: "재고" 관련 문서 찾기
grep -i "재고\|inventory\|stock" .claude/shared-references/notion-manifest.md
```

**매니페스트 검색 키워드 추출 규칙:**
1. 현재 기획 대상의 **도메인 키워드** (예: 재고, 생산, 판매)
2. 사용하는 **엔티티명** (예: Lot, Product, Partner)
3. 연결되는 **업무 프로세스명** (예: 입고, 출고, 조정)

### 2.3 plan-chatbot RAG 검색

```bash
cd tools/plan-chatbot

# 자연어 쿼리로 연관 문서 검색
python scripts/find_related_plans.py --query "재고 조정 Lot Warehouse" --top-k 10

# 특정 키워드 기반 검색
python scripts/find_related_plans.py --keywords "재고,Lot,입고,출고" --top-k 10

# 기존 문서 기반 연관 문서 검색 (문서 내용에서 키워드 자동 추출)
python scripts/find_related_plans.py --plan-id "abc123..." --top-k 10

# JSON 형식 출력 (프로그래밍 연동용)
python scripts/find_related_plans.py --query "재고 이동" --format json

# 특정 문서 제외 (자기 자신 제외)
python scripts/find_related_plans.py --query "재고" --exclude "현재문서_plan_id"
```

RAG 검색은 의미 기반이므로 정확한 키워드 일치가 아닌 문맥적 유사성으로 검색 가능.
출력에는 유사도(%), 매칭된 청크 수, 관련 섹션 헤딩이 포함됨.

### 2.4 Notion MCP 조회

```
# 식별된 문서의 상세 내용 확인 (필요시)
notion-fetch(page_id)
```

> **⚠️ Context Overflow 방지**: 연관 문서 조회는 최대 3건까지만 직접 fetch. 4건 이상은 subagent에 위임.

## 3. 정합성 검증 체크리스트

### 3.1 공유 엔티티 정합성

- [ ] **필드명 일치**: 동일 엔티티의 필드명이 문서 간 동일한지 (예: `quantity` vs `수량`)
- [ ] **상태 enum 일치**: 동일 엔티티의 상태 값 목록이 문서 간 동일한지
- [ ] **CRUD 제약 일치**: 수정/삭제 가능 조건이 문서 간 모순 없는지
- [ ] **데이터 타입 일치**: 같은 필드의 데이터 타입이 문서 간 동일한지

### 3.2 상태 전이 정합성

- [ ] **전이 연결 확인**: A 문서의 출력 상태가 B 문서의 유효한 입력 상태인지
- [ ] **역방향 전이**: 취소/되돌리기 시 연관 문서의 상태도 정의되어 있는지
- [ ] **동시 상태 모순**: 두 문서에서 동일 엔티티의 상태가 모순되는 경우가 없는지

### 3.3 네비게이션 정합성

- [ ] **양방향 연결**: A→B 연결이 있으면 B 문서에도 A와의 관계가 명시되어 있는지
- [ ] **파라미터 전달**: 화면 간 이동 시 전달되는 파라미터가 양쪽 문서에 명시되어 있는지

### 3.4 권한 정합성

- [ ] **권한 코드 일치**: 동일 리소스에 대한 권한 코드가 문서 간 동일한지
- [ ] **역할별 권한 범위**: 특정 역할이 A 기능은 가능하고 B 기능은 불가능할 때 논리적으로 타당한지

### 3.5 API 정합성

- [ ] **API 응답 필드 일치**: 같은 API를 사용하는 문서들이 동일한 응답 필드를 기대하는지
- [ ] **호출 시점 충돌**: 동일 API에 대한 호출 조건이 문서 간 충돌하지 않는지

### 3.6 데이터 파생 정합성

- [ ] **계산 로직 일치**: 집계/파생 데이터의 계산 방식이 원본 데이터 문서와 일치하는지
- [ ] **갱신 시점**: 파생 데이터가 언제 갱신되는지 명시되어 있는지

## 4. 충돌 처리 규칙

### 4.1 충돌 유형별 처리

| 충돌 유형 | 심각도 | 처리 방법 |
|----------|--------|----------|
| 엔티티 필드명 불일치 | Critical | 기존 확정 문서를 기준으로 통일, 변경 문서에 이력 기록 |
| 상태 enum 불일치 | Critical | 코드 구현 기준으로 통일, 모든 연관 문서 동시 업데이트 |
| 비즈니스 규칙 모순 | Critical | 사용자에게 결정 요청, 결정 근거를 설계 의도에 기록 |
| 권한 범위 불일치 | Major | 더 제한적인 쪽을 기준으로 통일 |
| UI 용어 불일치 | Minor | 최신 확정 문서를 기준으로 통일 |

### 4.2 충돌 보고 형식

```markdown
## 연관 문서 충돌 발견

| # | 충돌 유형 | 현재 문서 | 연관 문서 | 현재 값 | 연관 값 | 제안 |
|---|----------|----------|----------|--------|--------|------|
| 1 | 상태 enum | 재고 조정 | 재고 이동 | PENDING, COMPLETED | PENDING, IN_PROGRESS, COMPLETED | IN_PROGRESS 추가 검토 |
```

### 4.3 충돌 해결 후 조치

1. **양쪽 문서 동시 업데이트**: 한쪽만 수정하면 새로운 불일치 발생
2. **변경 이력에 교차 참조 기록**: "재고 이동 v2.1과 정합성 맞춤" 형식
3. **설계 의도에 결정 근거 기록**: 왜 이쪽 값을 선택했는지

```markdown
| # | 검증 항목 | 결과 | 상세 |

| 2 | 상태 enum | ⚠️ 불일치 | 현재: 3개, 연관: 4개 |

### 반영 사항
- [충돌 해결 내용 또는 추가 명시 필요 항목]
```
