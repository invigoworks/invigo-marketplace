## 작성 가이드라인

### 1. 변경 유형 표기

| 표기 | 의미 | 사용 시점 |
|------|------|----------|
| 추가 | 새로운 항목 추가 | 신규 화면, 필드, API, 로직 |
| 수정 | 기존 항목 변경 | 스펙 변경, 로직 변경 |
| 제거 | 기존 항목 삭제 | 불필요 기능 제거 |
| 보완 | 누락 사항 보충 | 기존 기획의 미비점 보완 |

### 2. 영향 표기

| 표기 | 의미 |
|------|------|
| FE | 프론트엔드만 영향 |
| BE | 백엔드만 영향 |
| Both | 프론트엔드 + 백엔드 모두 영향 |

### 3. 비교 형식 규칙

- **변경 전/후 비교**: 반드시 명시적으로 이전 상태와 변경 상태를 나란히 표기
- **코드 블록 사용**: 타입 정의, API 스펙 등은 코드 블록으로 표기
- **하이라이트**: 변경된 부분은 주석으로 표시 (예: `// 추가`, `// 변경`)

### 4. 체크리스트 우선순위

1. **Critical**: 기능 동작에 필수 (즉시 적용 필요)
2. **High**: 주요 기능 관련 (우선 적용)
3. **Normal**: 일반 변경 사항
4. **Low**: 개선 사항 (여유 시 적용)

## 변경 명세서 vs 기획서 업데이트

| 구분 | 기존 기획 업데이트 (Mode 3) | 기획 변경 명세서 (Mode 4) |
|------|---------------------------|--------------------------|
| **시점** | 기획 확정 전 | 기획 확정 후 |
| **목적** | 기획 내용 보완/추가 | 변경 사항 추적 및 전달 |
| **대상** | 기획자, 검토자 | 개발자 (FE/BE) |
| **포맷** | 기획서 본문 수정 | 변경 전/후 비교 형식 |
| **Notion 상태** | 기획 초벌 → 부서협의중 → 기획확정 | 기획확정 → 기획 변경 |
| **버전** | 1.0 → 1.1 → 2.0 | 2.0 → 2.1 → 2.2 |
