# 학습된 기획 규칙 (기획봇 피드백 기반)

> 기획봇 에스컬레이션에서 반복적으로 발견된 기획서 누락 패턴.
> 기획서 작성 시 이 체크리스트를 함께 참조하여 누락을 사전 방지합니다.

---

## 1. 데이터 모델 필수 명시 항목 (인사이트 #3)

기획서 PART 2 "데이터 명세" 작성 시, 코드에 구현된 필드가 기획서에 누락되는 패턴이 반복됨.

### 체크리스트

- [ ] **파생 필드 명시**: 다른 엔티티에서 참조하여 표시하는 필드(예: 납부기한, 거래처명)의 출처와 조인 관계를 명시했는가?
- [ ] **상태 enum 전체 나열**: 코드에 정의된 Status enum의 모든 값이 기획서에 표 형태로 정의되어 있는가? (예: PENDING, PARTIAL, COMPLETED, DELETED)
- [ ] **타입/인터페이스 필드 완전성**: mock 데이터나 타입 정의에 있는 모든 필드가 기획서 테이블 컬럼 또는 데이터 명세에 포함되어 있는가?

### 근거 사례
- 신고서 목록에 `납부기한` 필드가 코드에는 있으나 기획서에 미명시 → 기획자 혼란 (제안 #2, #4)
- 판매주문서의 `SalesOrderStatus` enum이 기획서에 미정의 → 필수 컬럼/validation 누락 (제안 #13)
- 생산입고의 Lot 관련 필드가 기획서에 미명시 → 창고 수정 가능 여부 판단 불가 (제안 #20)

---

## 2. 비즈니스 규칙 필수 명시 항목 (인사이트 #5)

코드에 구현된 비즈니스 로직이 기획서에 명시되지 않아 개발자/기획자 간 불일치 발생.

### 체크리스트

- [ ] **CRUD 동작별 제약 조건**: 수정/삭제가 가능한 상태 조건을 명시했는가? (예: "전송 전까지만 수정 가능", "확정 후 삭제 불가")
- [ ] **다단계 프로세스 분리**: 발주→출고, 신청→승인 등 2단계 이상 분리된 업무 흐름이 기획서에 명시되어 있는가?
- [ ] **수정 불가 필드 명시**: 등록 후 수정이 불가능한 필드(예: Lot 생성 후 창고 변경 불가)를 명확히 표시했는가?

- [ ] **수정 모드 재고 기준값 명시**: 등록된 데이터를 수정할 때 기준이 되는 값(조정 전 재고, 등록 시 재고 등)이 어느 시점의 스냅샷인지 명시했는가?
- [ ] **후속 트랜잭션 존재 시 수정 차단**: Lot/재고 관련 수정 시 해당 데이터에 후속 트랜잭션이 있으면 수정을 차단하는 규칙을 명시했는가?
- [ ] **신규/수정 모드 동작 분기**: 동일 화면에서 신규 등록과 수정 모드의 필드 동작 차이(예: 창고 변경 시 품목 초기화 여부)를 구분하여 명시했는가?

### 근거 사례
- 판매주문서가 발주 역할인지, 즉시 재고 차감인지 불명확 → 2단계 분리 구조 미명시 (제안 #15)
- 생산입고에서 Lot 생성 후 창고 수정 불가 규칙 미명시 (제안 #20)
- 주세신고서의 수정/삭제 가능 상태 조건 미명시 (제안 #1)
- 재고 조정 수정 시 "조정 전 재고"가 등록 시점 스냅샷인지 현재 재고인지 미명시 (제안 #32)
- 재고 이동 수정 시 출발 창고 변경 동작이 신규/수정 모드에서 다르게 처리되나 미명시 (제안 #34)

## 3. 컴플라이언스/증빙 필수 명시 항목 (인사이트 #4)

법적 서식, 세금 관계, 증빙 요건이 기획서에 누락되어 법적 정합성 확인 불가.

### 체크리스트

- [ ] **법적 서식 매핑**: 법정 제출 서식(별지 제N호)이 있는 경우, 서식의 각 필드가 시스템 어떤 필드에 매핑되는지 명시했는가?
- [ ] **세금/회계 레이어 분리**: 부가세(영세율)와 주세(미납세) 등 서로 다른 세금 레이어가 독립적으로 동작하는 경우, 각 레이어의 관계와 독립성을 명시했는가?

### 근거 사례
- 영세율과 미납세의 관계가 기획서에 미명시 → 자동 매핑 오해 발생 (제안 #6)
- 환입/폐기 신청서가 별지 제5호 서식에 매핑되어야 하는데 서식 필드 매핑 부재 (제안 #7)
